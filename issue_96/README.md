# Issue #96 — Race Condition in Pipelined Matmul

## Summary

When `T.Pipelined` is used with `num_stages > 0` in a matmul kernel, the
generated CUDA code introduces a race condition that can produce incorrect
results on GPUs such as RTX 4070 Ti and A100 for large matrix sizes
(e.g., N=8192).

---

## Bug Description

The `T.Pipelined` loop with `num_stages=3` generates a software-pipelined
kernel where multiple stages overlap:

- One stage computes the current tile
- Other stages prefetch future tiles into shared memory

This uses a multi-stage (triple-buffered) shared memory pipeline.

From the generated CUDA kernel, the loop structure looks like:
```bash
for (int ko = 0; ko < ...; ++ko) {
__syncthreads(); // sync before writing new data
... write stage (ko+2)%3 ... // prefetch next tile
__syncthreads(); // sync after writing
... mma using stage (ko%3) ... // compute current tile
}
```

---

## What Goes Wrong

The issue arises due to incorrect synchronization between pipeline stages.

- Shared memory buffers are reused across stages using modulo indexing
- A future stage writes into a buffer slot `(ko+2)%3`
- That same buffer slot may still be in use by computation from previous iterations

There is no guarantee that:
- all threads have finished reading from a shared memory slot
- before another stage starts overwriting it

This leads to a **write-after-read (WAR) hazard**, where:
- a buffer is overwritten while it is still being read

As a result, threads may read:
- stale data
- partially updated data
- incorrect values

---

## Root Cause

The synchronization (`__syncthreads()`) generated by TileLang is insufficient
to enforce proper ordering between:

- shared memory reads (during MMA computation)
- shared memory writes (during pipeline prefetch)

This causes a race condition across pipeline stages.

---

## Reproduction

```bash
python reproduce.py
```

## What to Look For

In the generated CUDA kernel:
* Identify the pipelined loop (for (int ko = ...))
* Observe that shared memory is reused across stages using modulo indexing
* Notice that:
  * writes to future stages happen before previous reads are guaranteed to complete
  * synchronization only surrounds writes, not full read-write ordering

This indicates a potential race between pipeline stages.

## Hardware Notes

* This issue is hardware-dependent and appears on GPUs such as:
  * RTX 4070 Ti
  * A100
* It requires:
  * large matrix size (e.g., N=8192)
  * pipelined execution (num_stages > 0)
* Test environment used here:
  * GPU: NVIDIA GeForce MX450 (SM75, 2GB VRAM)
  * Limitation: This GPU cannot execute the kernel due to lack of SM80 MMA support
(CUTE_ARCH_MMA_SM80_ENABLED error)
* However, the problematic CUDA code pattern is still reproducible via compilation.
